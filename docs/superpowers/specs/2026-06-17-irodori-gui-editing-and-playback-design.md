# Irodori GUI Editing And Playback Design

## Goal

次の未対応項目を、既存の `Irodori Studio` の操作文脈を崩さずに改善する。

- 行の番号指定による差し込み移動
- 特定参照音声カラムの生成結果一括消去
- 行削除 undo の残り猶予可視化
- セル内再生 UI の操作性改善

## Scope

### In Scope

- 各行に対するインラインの「移動先番号」操作 UI
- 既存 reorder API を使った番号指定移動
- マトリックスの各参照音声ヘッダーからの列消去操作
- 指定参照音声列の生成結果だけを消す backend API
- 右下 pending delete トーストへの円形 undo リング追加
- セル内 audio controls のサイズと配置見直し
- 補助操作ボタンの配置見直し
- 右側詳細ペインの再生 UI の余白・サイズ感の軽微な調整

### Out of Scope

- 独自オーディオプレイヤーの実装
- 参照音声そのものの削除フロー変更
- 行削除 undo の秒数表示
- レイアウト全体の再設計
- テーマカラー刷新
- 複数行同時移動

## Problem Summary

現状はドラッグ移動しか実質的な並べ替え手段がなく、行数が多い場合に遠距離移動が不便である。また、特定参照音声だけ結果を作り直したい時に列単位で生成結果を消す手段がなく、再生成準備の操作コストが高い。加えて、行削除 undo は猶予が見えず、いつ取り消せなくなるのかが分かりにくい。再生 UI はセル内の音声操作領域が小さく、補助操作も近接しているため押しづらい。

今回の修正では、行や列の近くで完結する局所的な操作 UI を採用し、既存の状態モデルや保存経路をできるだけ再利用しながら、編集と再生の操作性を上げる。

## Design Direction

### 1. Local Action UI

新しい操作は対象に近い場所へ置く。

- 行移動は各行の操作エリア
- 列消去は各参照音声ヘッダー
- undo 可視化は既存トースト
- 再生改善はセル内と詳細ペインの既存再生領域

これにより、別パネルやモーダル中心の管理 UI を増やさず、今のマトリックス中心ワークフローを保てる。

### 2. Reuse Existing Persistence Paths

行移動は新しい保存 API を増やさず、既存の reorder API に寄せる。逆に、列消去は既存 API では表現できないので、責務を限定した専用 API を1本追加する。undo リングと再生 UI はフロント責務に閉じる。

### 3. Incremental Playback Improvement

今回は独自プレイヤーへは進まず、ネイティブ `audio controls` を活かす。ブラウザ標準の再生・シーク互換性を保ちながら、セル内の見せ方と押しやすさを改善する。将来的な独自プレイヤー化に備え、セル内の領域分割だけは整理しておく。

## Feature Design

### Numbered Line Move

各行の操作エリアに `移動` ボタンを追加し、押した行だけインラインの移動 UI を展開する。

- 同時に開けるのは1行だけ
- 入力欄には `移動先番号` を表示する
- 入力値は `1..表示中行数` の範囲に制限する
- 実行は `Ctrl+Enter`
- キャンセルは `Esc` または `閉じる` ボタン
- 成功後は UI を閉じ、移動先の行を一時ハイライトする

意味づけは「指定した番号が移動後の最終行番号になる」で固定する。たとえば `5` を指定したら、その行は移動後に `5行目` になる。

### Move Interaction Details

- 他の行で `移動` を開いた場合、現在開いている行の UI は閉じる
- ドラッグ移動と番号移動は同じ reorder 保存経路を使う
- 範囲外エラーは実行時に出さず、そもそも入力時点で不正値を入れにくくする
- ハイライトは短時間だけ残り、その後通常表示へ戻る

### Reference Column Clear

各参照音声ヘッダーに `この列を消去` ボタンを追加する。

- 押下時に確認ダイアログを出す
- OK 時のみ実行する
- 対象はその参照音声列に属するセルの「生成結果」だけ
- 参照音声定義自体は残す
- 成功後のセル表示は通常の `未生成` に戻る

`列消去済み` の特別状態は増やさない。既存の `not_generated` 表示へ素直に戻す。

### Undo Ring

既存の pending delete トーストに円形リングを追加する。

- テキスト秒数は出さない
- リングの減少だけで残り猶予を示す
- 既存の undo ボタンや文言は維持する
- バックエンド状態は増やさない

リングはトーストの補助情報として扱い、主役は引き続き undo 操作そのものとする。

### Playback UI

セル内の構成を次の3段に整理する。

- 上段: 状態表示
- 中段: 大きめの `audio controls`
- 下段: 補助操作

補助操作は `＋リスト` と `再生成` を横並びの小ボタンとして整理する。ただし誤操作を避けるため、ボタン同士と再生 UI との間に十分な余白を取る。ホバー時だけ表示する方式にはせず、常時見えるが押しやすい配置にする。

右側詳細ペインは再設計しないが、セル内と極端に乖離しないよう、audio 要素の高さや上下余白を軽くそろえる。

## Architecture

### Frontend

主な変更対象は次の通り。

- `LineMatrix.tsx`
  - 行移動 UI の追加
  - 列消去ヘッダー操作の追加
  - 再生 UI レイアウト変更
  - 移動後ハイライト制御
- `ProjectEditor.tsx`
  - 列消去ハンドラ受け渡し
- `PendingDeleteToast.tsx`
  - 円形 undo リング描画
- `styles.css`
  - 移動 UI、列ヘッダー操作、undo リング、再生 UI のスタイル調整

### Backend

列消去専用 API を追加する。

- 入力: `project_id`, `reference_id`
- 作用: 対象 reference に属する全セル結果の消去
- 非作用:
  - 他 reference のセル
  - line 定義
  - reference 定義そのもの

Project のレスポンス形式は基本維持し、成功後の cell state 解決が既存ロジックで `not_generated` になるようデータを更新する。

## Data Flow

### Line Move

1. ユーザーが行の `移動` を押す
2. その行だけ移動入力欄が開く
3. 入力中に許容範囲内の番号へ制限する
4. `Ctrl+Enter` でフロントが新しい line ID 配列を生成する
5. 既存 reorder API へ送る
6. 成功したら移動 UI を閉じ、移動先行を一時ハイライトする

### Column Clear

1. ユーザーが列ヘッダーの `この列を消去` を押す
2. 確認ダイアログで承認する
3. 専用 API を呼ぶ
4. backend が対象列セルの生成結果だけを消す
5. 更新済み project を返す
6. frontend は通常更新として再描画し、対象列セルは `未生成` として見える

### Undo Ring

1. 行削除で pending delete が発生する
2. トーストが締切時刻を元に残り比率を計算する
3. requestAnimationFrame または短い interval でリング描画を更新する
4. 猶予が切れたらトーストごと消える

### Playback UI

1. 既存 audio 要素をそのまま使う
2. CSS と DOM 配置だけを見直す
3. 再生イベントや自動再生ロジックは既存フックを維持する

## Error Handling

### Line Move

- reorder API が失敗した場合は行順を確定変更しない
- 入力制限により範囲外エラーは避ける
- 表示中行数が変化した場合は、その時点の行数を基準に入力上限を更新する

### Column Clear

- API 失敗時は project をローカルで楽観更新しない
- 確認ダイアログを通るため、誤操作率は抑える
- 対象 reference が存在しない場合は backend で 404 または validation error を返す

### Undo Ring

- タイマー描画が乱れても undo 機能自体は壊さない
- リングは表示補助なので、描画更新失敗時もトースト本文と undo ボタンは機能させる

### Playback UI

- ネイティブ controls を維持するため、再生・シーク互換性はブラウザへ委ねる
- レイアウト調整で selection mode や既存クリック挙動を壊さない

## Testing Strategy

### Frontend

- `移動` UI が各行で開閉できること
- 同時に1行しか開かないこと
- `Ctrl+Enter` で reorder ハンドラが発火すること
- 実行成功後に UI が閉じ、移動先行へ一時ハイライトが付くこと
- 列ヘッダーの `この列を消去` から確認ダイアログを経て API 呼び出しされること
- 列消去後のセルが `未生成` 表示になること
- pending delete トーストに円形リングが表示されること
- 再生 UI のレイアウトクラスと補助操作配置が期待どおりであること

### Backend

- 列消去 API が対象 reference 列のセル結果だけを消すこと
- 他 reference 列のセル結果を壊さないこと
- reference 定義自体は残ること
- 返却 project の対象セルが `not_generated` 相当になること

## Risks

- 行移動 UI を行ごとに足すことで、行操作エリアが過密になる可能性がある
  - ボタン数を増やしすぎず、インライン UI は必要時だけ開く
- 列消去は破壊的操作なので、ボタン位置次第で誤押下が起きうる
  - 確認ダイアログを必須にする
- undo リングは見た目のためのタイマーであり、描画方式によってはテストが不安定になる
  - 描画値を純関数化するか、時間注入可能な構造に寄せる
- 将来独自プレイヤー化する際に DOM 構成が再変更になる
  - 今回はレイアウト責務だけ分離しておき、再生ロジックを触りすぎない

## Recommendation

今回の改善は、行・列・セルの近くに局所操作を追加し、保存経路は既存実装を優先的に再利用するのが最適である。

- 行移動は既存 reorder API を使う
- 列消去は専用 API を1本だけ追加する
- undo リングと再生 UI はフロント責務に閉じる
- 再生体験はネイティブ controls を活かしたレイアウト改善でまず確実に前進させる

これにより、今回の不満点を比較的小さな責務追加で解消しつつ、将来の拡張余地も残せる。
